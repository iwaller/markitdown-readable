# DOCX 转可读 Markdown 方案

本文说明 DOCX 转 Markdown 的处理方案。

转换目标不是还原 Word 的视觉排版。转换目标是保留文档语义。输出内容应明确、稳定，便于检索、规则处理和模型读取。

## 目标

- 保留段落、标题、列表、表格、图片、批注和修订。
- 将批注等上下文放在原文附近。
- 将复杂表格展开成逻辑行和逻辑列。
- 使用明确的语义标记，减少对样式的依赖。

## 输出原则

简单结构使用普通 Markdown。

- 段落输出为 Markdown 段落。
- 简单表格输出为 Markdown 表格。
- 图片使用 Markdown 图片语法。
- 标题尽量保留原有层级。

Word 专有语义使用显式标记。

```text
[DELETED: 删除内容]
[INSERTED: 新增内容]
[COMMENT-1 针对目标: 作者 日期: 批注内容]
```

显式标记比样式更稳定。例如，`[DELETED: ...]` 比 `~~...~~` 更明确。脚注也可能在分块时脱离正文。

## 批注处理

批注应放在被批注内容附近。

推荐格式如下。

```markdown
甲方应在 15 日内完成付款。
[COMMENT-1 针对此句: 张三 2026-06-26: 确认这里按自然日还是工作日计算。]
```

不要只把批注集中放到文末。文档分块后，正文可能无法关联到文末批注。

## 修订处理

删除文字使用 `[DELETED: ...]`。新增文字使用 `[INSERTED: ...]`。

推荐格式如下。

```markdown
甲方应在 [DELETED: 7 日内] [INSERTED: 15 日内] 完成付款。
```

不推荐只使用删除线。

```markdown
甲方应在 ~~7 日内~~ 15 日内完成付款。
```

删除线适合人工阅读，但语义不够明确。

## 表格处理

表格转换应优先保留逻辑含义。不要优先还原视觉布局。

### 简单表格

简单表格可以直接输出为 Markdown 表格。

```markdown
| 产品 | 规格 | 价格 |
|---|---|---|
| 面膜 | 5 片/盒 | 39 元 |
| 精华 | 30 ml | 99 元 |
```

### 合并数据单元格

数据区的合并单元格应展开。每一行都应包含完整信息。

推荐格式如下。

```markdown
| 类别 | 字段显示名 | 字段类型 |
|---|---|---|
| 模板 | 状态 | 布尔类型 |
| 模板 | 名称 | 文本 |
| 模板 | 创建人 | 文本 |
```

不推荐保留空白继承单元格。

```markdown
| 类别 | 字段显示名 | 字段类型 |
|---|---|---|
| 模板 | 状态 | 布尔类型 |
|  | 名称 | 文本 |
|  | 创建人 | 文本 |
```

空白单元格适合人工理解，但会丢失行级语义。

### 多级表头

合并表头应展开为明确的列名。

原始 Word 表格可能包含以下结构。

```text
字段显示名 | 字段类型 | 数据规则（横跨 3 列） | 示例值 | 备注
                        数据来源 | 是否必填 | 规则
```

推荐输出如下。

```markdown
| 字段显示名 | 字段类型 | 数据规则.数据来源 | 数据规则.是否必填 | 数据规则.规则 | 示例值 | 备注 |
|---|---|---|---|---|---|---|
| 状态 | 布尔类型 | 手动设置 |  | 默认激活，根据手动设置决定 | 激活 | 只有激活状态的模板才能被选择。 |
```

展开规则如下。

- 横向合并表头作为父级前缀。
- 子级表头使用 `.` 连接。
- 纵向合并表头保留原名。
- 数据区的合并值填充到每个受影响单元格。

### 复杂表格

复杂表格可以同时输出两种形式。

- HTML 表格保留 `rowspan` 和 `colspan`。
- Markdown 表格保留展开后的逻辑结构。

下游处理应优先使用展开后的 Markdown 表格。

## DOCX 解析范围

DOCX 本质是 ZIP 包。转换器需要直接读取 OOXML。这样可以避免通用转换丢失结构。

需要读取的文件如下。

```text
word/document.xml
word/comments.xml
word/_rels/document.xml.rels
word/numbering.xml
word/styles.xml
```

需要重点处理的元素如下。

```text
w:tbl
w:tr
w:tc
w:gridSpan
w:vMerge
w:del
w:delText
w:ins
w:t
w:commentRangeStart
w:commentRangeEnd
w:commentReference
```

如果通用转换已经丢失合并结构、批注或修订，后处理很难恢复这些信息。因此，DOCX 转换应直接解析 OOXML。

## 推荐架构

项目继续使用 MarkItDown 作为转换框架。项目为需要深度处理的格式注册专用转换器。

当前分工如下。

- DOCX：使用自定义 OOXML 转换器。
- PDF：提取文本，并输出页面级图片索引。
- PPTX：提取幻灯片、文本、表格、图表和图片。

DOCX 转换器负责以下能力。

- 合并表格展开。
- 多级表头扁平化。
- 批注读取和定位。
- 新增文字和删除文字标记。
- 图片和关系文件解析。
- 编号和标题样式推断。

## 总结

可读 Markdown 应让每一行、每一句、每张图片、每条批注和每处修订都能独立理解。

DOCX 转换应遵守以下规则。

- 展开合并单元格。
- 扁平化多级表头。
- 将批注放在目标内容附近。
- 使用 `[DELETED: ...]` 和 `[INSERTED: ...]` 标记修订。
- 在通用转换会丢失结构时，直接读取 OOXML。
