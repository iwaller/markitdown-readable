大模型通常会理解为：

但是如果文档复杂，尤其是法律合同、需求文档、报价单、招标文件，我更建议用更显式的格式，不要完全依赖 ~~ ~~ 和脚注。

1. 删除文字：建议用更明确的标记

~~删除文字~~ 大模型能懂，但它也可能把它当成“样式”而不是“修订语义”。

更稳的写法是：

这是原文，[DELETED: 这里是被删除的文字]，这是保留文字。

或者：

这是原文，<deleted>这里是被删除的文字</deleted>，这是保留文字。

如果还有新增文字，建议这样：

原句：
甲方应在 [DELETED: 7日内] [INSERTED: 15日内] 完成付款。

这种对大模型更清楚。

比起：

甲方应在 ~~7日内~~ 15日内 完成付款。

后者也能看懂，但没有前者稳定。

2. 批注 / 备注：脚注能懂，但最好靠近原文

[^comment-1] 这种脚注，大模型能理解。

但是如果你是做 RAG、分块读取、向量检索，有一个问题：正文和脚注可能被切到不同 chunk 里。这样模型看到正文时，可能看不到对应备注。

所以更推荐把备注直接放在附近：

甲方应在15日内完成付款。  
[COMMENT by 张三 at 2026-06-26: 这里需要确认是否按工作日计算。]

或者：

甲方应在15日内完成付款。 <!-- COMMENT: 张三：这里需要确认是否按工作日计算 -->

如果备注很长，可以正文里放短注，后面再集中列完整批注：

甲方应在15日内完成付款。[COMMENT-1]

[COMMENT-1]: 张三：这里需要确认是否按工作日计算。

对于大模型来说，最稳的是：

甲方应在15日内完成付款。
[COMMENT-1 针对此句: 张三：这里需要确认是否按工作日计算。]
3. 表格：普通 Markdown 表格可以，但合并单元格要特别处理

普通表格，大模型很容易理解：

| 产品 | 规格 | 价格 |
|---|---|---|
| 面膜 | 5片/盒 | 39元 |
| 精华 | 30ml | 99元 |

但 Word 里的合并单元格，如果直接转成普通 Markdown，信息容易丢。

比如原表格是：

类别	产品	价格
护肤	面膜	39
护肤	精华	99

Word 里可能是“护肤”这个单元格纵向合并了两行。

给大模型读时，最推荐把合并单元格展开：

| 类别 | 产品 | 价格 |
|---|---|---|
| 护肤 | 面膜 | 39 |
| 护肤 | 精华 | 99 |

不要为了还原视觉效果而写成：

| 类别 | 产品 | 价格 |
|---|---|---|
| 护肤 | 面膜 | 39 |
|  | 精华 | 99 |

因为模型可能不知道第二行的空白其实还是“护肤”。

4. 如果表格结构复杂，HTML table 比普通 Markdown 更合适

    <td>面膜</td>
    <td>39元</td>
  </tr>
  <tr>
    <td>精华</td>
    <td>99元</td>
  </tr>
</table>

大模型通常也能理解 rowspan、colspan。

但如果你后续要做信息抽取，我更推荐额外加一份“展开后的表格”：

原始表格结构：包含合并单元格。

展开后的逻辑表格：

| 类别 | 产品 | 价格 |
|---|---|---|
| 护肤 | 面膜 | 39元 |
| 护肤 | 精华 | 99元 |

也就是说，给人看用 HTML table，给模型理解用展开表格。

这种格式比单纯：

更适合大模型长期稳定读取。

6. 最推荐的处理原则

你可以按这个优先级来转：

内容类型	推荐给大模型的格式
普通段落	Markdown 段落
普通表格	Markdown table
合并单元格表格	展开成逻辑表格，必要时附原始 HTML table

脚注	可用，但 RAG 场景最好靠近正文

所以答案是：能理解，但建议你不要只转成“漂亮 Markdown”，而要转成“语义明确的 Markdown”。

[DELETED: 删除内容]
[INSERTED: 新增内容]
[COMMENT-1 针对此句: 批注内容]

而不是只用：

~~删除内容~~
[^comment-1]

这样模型理解会更稳定，尤其适合合同、需求文档、报价单、表格类文档。

from pathlib import Path
from markitdown import MarkItDown

def convert_for_llm(path: str) -> str:
    suffix = Path(path).suffix.lower()

    if suffix == ".docx":

        result = md.convert(path)
        return result.text_content

DOCX 这部分应该自己解析什么？

DOCX 本质是 zip 包。你需要读这些文件：

word/document.xml       # 正文、表格、修订标记
word/comments.xml       # 批注
word/_rels/document.xml.rels

你要重点解析：

w:tbl       表格
w:tr        行
w:tc        单元格
w:gridSpan  横向合并
w:vMerge    纵向合并

w:del       删除内容
w:delText   被删除文字
w:ins       新增内容
w:t         普通文本

w:commentRangeStart
w:commentRangeEnd
w:commentReference

最终输出你自己的 LLM Markdown：

|---|---|---|---|---|---|---|
| 状态 | 布尔类型 | 手动设置 |  | 默认激活，根据手动设置决定 | 激活 | 注：只有激活状态的模板才能在智能质检路由及人工计划模板中被选择；当启动切换为禁用时，提示当前被选中，不支持切换。 |
